# Продуктовый аудит Telegram-бота (черновик на согласование)

Дата: 2026-07-15. Скоуп: `backend/app/bot/` + пользовательские пути через
`app/ai/` и портал. Метод: разбор кода всех сценариев, сверка с UX-практиками
Telegram-ботов (командное меню, инлайн-кнопки, обработка ошибок) и с флоу
зрелых продуктов родительского контроля (Google Family Link, Apple Screen
Time — паттерн «запрос ребёнка → одобрение родителя»).

## 1. Что уже сильно (не разрушать)

- **Guardrails**: allowlist chat_id; мутации ИИ строго через кнопку;
  реестр tools как единственная точка входа; self-IP защита; дневной
  бюджет токенов. Это лучше большинства «домашних» ботов.
- Кнопочные уведомления о новых устройствах (назначить/блок) — правильный
  паттерн: действие в один тап в момент события.
- Health-алерты с антидребезгом; деградация без ИИ-ключа.
- Тонкие хендлеры поверх tools — команды, ИИ и панель используют одну логику.

## 2. Главные находки

### Продукт: бот реактивен там, где должен быть проактивен

| # | Находка | Следствие |
|---|---|---|
| P1 | **Блокировка по квоте происходит молча.** `NOTIFY_KINDS` — только device_new / register_request / maybe_same; событие `block` не рассылается | Ребёнок «остался без интернета», родитель узнаёт от ребёнка, а не от бота. Доверие к системе падает |
| P2 | **Нет флоу «попросить время»** — киллер-фича Family Link/Screen Time. Страница /blocked пассивно советует «попроси родителей» | Родитель вынужден вручную набирать `/bonus Планшет 30 games`. Самый частый семейный сценарий — самый неудобный |
| P3 | Квоты и экранное время не видны в `/status`, а **в дайджест не попали вовсе** (в ТЗ фазы 2 строка «экранное время за день» заявлена — не реализована) | Родитель не видит главного показателя без запроса к ИИ |
| P4 | Подсказки об объединении MAC-дублей только реактивные (в момент появления двойника) | 20 устройств с «приватными» MAC копятся; нет команды «покажи подозрения на дубли» |

### UX: команды вместо кнопок, текст вместо выбора

| # | Находка | Следствие |
|---|---|---|
| U1 | **`set_my_commands` не вызывается** — у бота нет командного меню Telegram | Нулевая обнаруживаемость: команды надо помнить наизусть; /help — простыня |
| U2 | `/block <имя>` при **неоднозначном** имени отвечает «Не нашёл устройство» (find_device возвращает None и при 2+ совпадениях) | Ложное сообщение; правильный ответ — кнопки-кандидаты |
| U3 | Мутации по текстовому вводу без выбора: `/pause kid 60`, `/bonus Миша 30 games` — синтаксис из справки | Порог входа для второго родителя; ошибки ввода |
| U4 | `/devices` — все 20 устройств одной простынёй, без онлайн/оффлайн (данные есть в `get_online_ips`, но в `list_devices` не пробрасываются), без квот, без группировки по людям | Список нечитаем ровно тогда, когда нужен |
| U5 | Нет пресетов частых действий: «пауза детям на час», «до утра», «ужин» | Каждое действие — ручной ввод |
| U6 | Кнопки подтверждения ИИ-мутаций **умирают при каждом деплое** (PENDING_ACTIONS в памяти процесса, а CD рестартит бота на каждый push) | «Действие устарело — повторите запрос» как регулярный опыт |
| U7 | Ошибки вида «Что-то пошло не так, подробности в журнале бота» | Родителю журнал недоступен из бота (только через ИИ) |

### Функциональность: недостающие сценарии

- Нет `/quota` (прогресс дня без ИИ) и `/events` (журнал без ИИ).
- Нет предупреждения «квота на 80%» — только жёсткая отсечка.
- Нет weekly-отчёта (у Screen Time — недельные тренды).
- Час дайджеста зашит константой (21:00), не настраивается.
- Нет `/usage` — расход ИИ-бюджета (данные уже пишутся в kv_state).
- Голосовые сообщения не обрабатываются (частый мобильный сценарий).

### Архитектура и надёжность

| # | Находка | Риск |
|---|---|---|
| A1 | **Два процесса пишут в RouterOS**: reconcile зовут и планировщик панели, и бот после каждой мутации | Гонка address_list_sync → дубли записей в hs-списках; лишняя связность (бот знает креды роутера) |
| A2 | `notify.py` восстанавливает устройство **регексом по MAC из текста** журнала | Хрупкая связь: правка текста события ломает уведомления молча |
| A3 | Состояние бота в памяти: PENDING_ACTIONS, CHAT_HISTORY, счётчик pending | Теряется на каждом деплое (см. U6) |
| A4 | Дайджест-петля — свой sleep-цикл в боте, у панели — APScheduler | Два механизма расписаний; время не конфигурируется |
| A5 | Роли не различаются: все chat_id — полные админы | Нельзя дать бабушке «только смотреть», ребёнку — «только просить» |

## 3. Предложения (приоритизированы)

### Спринт 1 — гигиена UX (1 PR, всё S/M)

1. **Меню команд**: `bot.set_my_commands()` при старте + короткий /start
   с 3 кнопками («Статус», «Устройства», «Пауза детям…»). *(S)*
2. **Persist PENDING_ACTIONS** в таблицу (id, tool, args, description,
   created) — кнопки переживают деплой; чистка по TTL сутки. *(M)*
3. **Выбор устройства кнопками** везде, где сейчас `<имя|id>`: неоднозначное
   имя → инлайн-список кандидатов; `/block` без аргумента → список. *(M)*
4. **`/devices` v2**: группировка по людям, метки 🟢/⚪ онлайн (пробросить
   online в list_devices), ⛔/⏸/🐢 уже есть; при >12 устройств — постранично. *(M)*
5. **Уведомление о сработавшей квоте/блокировке** (kind `block` в
   NOTIFY_KINDS + кнопка «+30 мин»). *(S)*
6. `/status` + строка экранного времени по детям; дайджест + то же
   (дореализация ТЗ фазы 2). *(S)*

### Спринт 2 — киллер-фича: запрос времени (1 PR)

7. **Кнопка «Попросить ещё времени» на /blocked** (только для квотных
   блокировок): создаёт BonusRequest → бот шлёт родителю пуш с кнопками
   [+15] [+30] [до конца дня] [Отказать] → add_bonus_time. Rate-limit как
   у портала (одна просьба в 10 минут). Родитель отвечает в один тап,
   ребёнок видит результат на той же странице (автообновление). *(M/L)*
8. **Пресеты-кнопки**: «⏸ 30м / 1ч / до утра» под /pause без аргументов;
   «Ужин» = пауза kid 30. *(S)*
9. `/merge` — список подозрений на дубли (random-MAC + совпадение
   hostname/имени) с кнопками объединения. *(M)*

### Спринт 3 — архитектура (1 PR)

10. **Один писатель в RouterOS**: мутации бота меняют БД и ставят флаг
    `reconcile_asap` в kv_state; планировщик панели проверяет его каждые
    5 секунд лёгким запросом. Бот перестаёт ходить в MikroTik. *(M)*
11. **Outbox вместо регекса**: колонка device_id в EventLog (аддитивная
    миграция №2) — notify.py читает структурно. *(M)*
12. Настройки: HS_DIGEST_HOUR, HS_QUOTA_WARN_PCT (80), роли
    HS_TELEGRAM_VIEWER_IDS (только чтение). *(S)*
13. `/usage` — расход ИИ-бюджета за сегодня/неделю. *(S)*

### Отложить осознанно

- **Mini App / веб-панель в Telegram** — требует публичный HTTPS, панель
  за NAT; не делать, пока не появится туннель (и он сам по себе риск).
- **Голосовые команды** — транскрипция требует внешнего API; вернуться,
  когда появится дешёвый путь.
- **Автономные действия ИИ без кнопки** — не ослаблять принцип №1 ТЗ.
  Единственное допустимое исключение обсуждать отдельно (автопауза
  «тихих часов» как opt-in правило, а не решение ИИ).

## 4. Метрики успеха (простые, по журналу)

- Доля мутаций через кнопки vs текстовый ввод (растёт → UX работает).
- Число «Действие устарело» после деплоев (должно упасть до ~0 после №2).
- Время от квотной блокировки до реакции родителя (после №5/№7).
- Запросы `+время` через страницу vs устные (приживаемость киллер-фичи).
