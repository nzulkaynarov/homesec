# ТЗ (черновик на согласование): рефакторинг + ИИ-мультиагентный слой (Opus)

Статус: **черновик, ждёт согласования владельцем**. Ничего из описанного ещё не
реализовано. После согласования каждый этап идёт отдельной веткой/PR, в `main`
попадает только зелёное (CD: push в `main` = деплой в прод).

## 1. Цели

1. Подготовить кодовую базу к безопасной работе ИИ-агентов — и как
   разработчиков репозитория, и как runtime-фич продукта.
2. Реализовать ИИ-слой на Claude Opus: несколько специализированных агентов
   (оркестратор NL-команд, аналитик-дайджест, сторож-аномалий) поверх общего
   реестра инструментов (tool registry).
3. Закрыть Фазу 1 дорожной карты (Telegram-бот) — он обязательная транспортная
   основа для ИИ-фич (уведомления, подтверждения действий, NL-канал).

Не-цели: перевод Huawei в мост, замена веб-панели, JS-фреймворки, изменение
firewall/NAT/mangle-логики (`.rsc` остаётся единственным источником правил).

## 2. Краткий аудит текущего состояния

Сильные стороны: компактное идемпотентное ядро (`enforcement.reconcile`),
чистое разделение сервисов, «выстраданные правила» задокументированы, тесты
не требуют живой сети.

Найденные проблемы (кандидаты на рефакторинг):

| # | Проблема | Где | Риск |
|---|---|---|---|
| 1 | Синхронный I/O (librouteros, httpx) вызывается прямо в `async`-хендлерах — блокирует event loop; при недоступном роутере панель «замирает» на таймаут для всех | `routers/*.py`, `services/*` | средний |
| 2 | Нет миграций схемы: `create_all` не добавляет колонки в существующие таблицы — любое новое поле в прод-базе требует ручного ALTER на малинке | `db.py`, `main.py` | высокий для этапов ниже |
| 3 | SQLite без WAL — два процесса (панель + будущий бот) будут ловить `database is locked` | `db.py` | высокий для этапа 1 |
| 4 | Нет линтера/type-check в CI | `.github/workflows/ci.yml` | низкий |
| 5 | `settings` читается при импорте — тесты вынуждены хакать env в `conftest` до импорта | `config.py` | низкий |
| 6 | `EventLog` растёт без ротации | `models.py` | низкий |
| 7 | `PUBLIC_PATHS` сравнивается через `startswith` — путь `/login-x` тоже публичный (безвредно, но неаккуратно) | `auth.py` | косметика |

## 3. Целевая архитектура ИИ-слоя

```text
Telegram (long polling, исходящее — обходит NAT)
     │
homesec-bot (отдельный systemd-юнит, тот же venv и SQLite-база в WAL)
     │  уведомления, кнопки-подтверждения, /status /block /pause, NL-текст
     ▼
backend/app/ai/
  ├─ client.py     — anthropic SDK, модель из HS_AI_MODEL, бюджет-guard
  ├─ tools.py      — реестр инструментов: типизированные обёртки над
  │                  services/* (единственная точка, где ИИ трогает систему)
  ├─ orchestrator.py — агент NL-команд: текст из Telegram → tool calls
  ├─ analyst.py    — ежедневный дайджест (querylog AdGuard + EventLog)
  └─ watchdog.py   — детектор аномалий (ночная активность, всплеск DoH,
                     health: AdGuard/панель/роутер упали)
```

Принципы безопасности (жёсткие, зашиты в код `tools.py`, не в промпт):

1. **Read-only по умолчанию.** Мутирующие инструменты (`block_device`,
   `pause_internet`, `set_speed_limit`) исполняются ТОЛЬКО после нажатия
   кнопки подтверждения в Telegram. Агент готовит действие — человек жмёт.
2. ИИ работает только через существующие пути enforcement (address-lists
   `hs-*`, очереди `hs-dev-*`, клиенты AdGuard `hs-*`). Никаких инструментов
   для firewall/NAT/mangle/DNS-конфига — их физически нет в реестре.
3. Защита `get_self_ips()` действует и для ИИ: малинку заблокировать нельзя.
4. Лимиты: N tool-calls на запрос, дневной бюджет токенов (`HS_AI_DAILY_BUDGET`),
   при исчерпании — вежливый отказ, не деградация.
5. Все действия и решения ИИ — в `EventLog` (kind `ai_*`): полный аудит.
6. Telegram-доступ только для chat_id из allowlist (`HS_TELEGRAM_CHAT_IDS`).
7. Деградация без интернета/API: панель и enforcement работают как сейчас,
   ИИ-фичи просто молчат (и это отражено в /status).

Модели: по умолчанию `claude-opus-4-8` для оркестратора и аналитика
(качество рассуждений), опционально дешёвая модель для рутинной классификации
(«это аномалия или нет?») — настраивается через env, вопрос №2 ниже.

## 4. Этапы работ

### Этап 0 — фундамент (рефакторинг, поведение не меняется)

- `ruff` (lint + format check) и `mypy` (мягкий режим) в CI; фикс находок.
- SQLite: включить WAL + `busy_timeout` (подготовка ко второму процессу).
- Мини-миграции: таблица `schema_version` + идемпотентные ALTER'ы при старте
  (Alembic для проекта такого размера — избыточен).
- Убрать блокировку event loop: сервисные вызовы в хендлерах — через
  `run_in_threadpool` (точечно, без переписывания сервисов на async).
- `tools.py`: реестр типизированных операций поверх services/* с докстрингами,
  из которых генерируются tool-схемы для Claude. Панель постепенно переводится
  на те же операции (одна точка входа для человека и ИИ).
- Ротация EventLog (хранить 90 дней), фикс `PUBLIC_PATHS`.
- Тесты на tools-слой. Ожидаемый объём: 1 PR, ~400–600 строк diff.

### Этап 1 — Telegram-бот (Фаза 1 дорожной карты)

- Библиотека: `aiogram` 3.x. Отдельный systemd-юнит `homesec-bot`
  (переживает рестарты панели и умеет алертить, если панель упала).
- Уведомления: новое устройство (кнопки: «в гости» / «ребёнку…» / «блок»),
  health-алерты (AdGuard, панель, роутер недоступны — с антидребезгом).
- Команды: `/status`, `/block <устройство>`, `/unblock`, `/pause <группа> <мин>`.
- `.env`: `HS_TELEGRAM_BOT_TOKEN`, `HS_TELEGRAM_CHAT_IDS`.
- deploy: юнит + правка `install.sh`/`update.sh` (рестарт обоих сервисов).
- Ожидаемый объём: 1–2 PR.

### Этап 2 — ИИ-агенты (Opus)

- `ai/client.py` + бюджет-guard + учёт расхода токенов в EventLog.
- **Analyst**: по расписанию (21:00) — дайджест дня в Telegram: топ-домены
  по устройствам детей, заблокированные попытки, новые устройства, здоровье.
- **Watchdog**: каждые 15 мин по дешёвым эвристикам (без LLM) ищет триггеры —
  ночная активность устройства ребёнка, всплеск попыток DoH; при срабатывании
  LLM формулирует человеческий алерт с контекстом.
- **Orchestrator**: свободный текст в Telegram → tool calls
  («заблокируй планшет на час», «что сегодня смотрел Миша?», «сними лимит»).
  Мутации — только через кнопку подтверждения (принцип №1).
- `.env`: `HS_ANTHROPIC_API_KEY`, `HS_AI_MODEL`, `HS_AI_DAILY_BUDGET`.
- Ожидаемый объём: 2–3 PR (analyst+watchdog отдельно от orchestrator).

### Этап 3 (опция, Фаза 2 дорожной карты) — после обкатки этапа 2

Квоты времени по категориям, страница «время вышло», портал регистрации
неизвестных устройств, эвристики против MAC-рандомизации (сопоставление по
hostname + вендору + времени появления). Детализируем отдельным ТЗ.

## 5. Подготовка репозитория к работе ИИ-агентов-разработчиков

- CLAUDE.md уже хорош; добавить короткие модульные докстринги там,
  где появятся ai/ и bot/.
- CI обязателен зелёный до merge в main (уже так); добавить lint-гейт (этап 0).
- Рискованные этапы (1, 2) обкатываются на ветке: панель+бот можно поднять на
  малинке вторым портом/токеном для проверки без деплоя в main.

## 6. Риски

| Риск | Митигируется |
|---|---|
| Ошибка в новых зависимостях роняет панель при CD | requirements с пином версий, тесты импорта, поэтапные PR |
| Pi 3B (1 ГБ RAM) — два python-процесса | aiogram+anthropic легковесны; смотреть RSS в /status |
| Стоимость Opus | дневной бюджет-guard, дайджест 1 раз/день, watchdog без LLM до триггера |
| ИИ-галлюцинация → неверная блокировка | подтверждение кнопкой, аудит, self-IP защита |
| Telegram-токен = удалённое управление домом | allowlist chat_id, мутации только с подтверждением |
