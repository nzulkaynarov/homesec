# ТЗ (черновик на согласование): Фаза 2 — квоты времени, «время вышло», портал, MAC-рандомизация

Статус: **черновик, ждёт согласования владельцем**. Реализация — после ответов
на вопросы §7. База: этапы 0–2 (реестр инструментов, бот, ИИ) уже в ветке.

## 1. Цели

1. **Квоты времени**: «детям — 2 часа игр в день», «планшету — 3 часа
   интернета в день»; по исчерпании — блокировка до полуночи, с возможностью
   бонуса («+30 минут») из Telegram/ИИ.
2. **Страница «время вышло»**: заблокированное устройство видит понятную
   страницу с причиной и остатком, а не молчаливо «сломавшийся интернет».
3. **Портал регистрации**: неизвестное устройство само представляется
   («это телефон бабушки») — владельцу остаётся нажать кнопку в Telegram.
4. **Анти-MAC-рандомизация**: телефон с «приватным Wi-Fi адресом» не плодит
   бесконечных «новых устройств», а распознаётся и объединяется со старым.

Не-цели: подсчёт трафика в байтах, DPI/анализ HTTPS, приложение на телефон,
изменение архитектуры enforcement.

## 2. Квоты времени

### Как меряем «время»

Точной телеметрии с устройств нет — считаем по **DNS-активности** (журнал
AdGuard уже есть). Минута считается «активной», если устройство сделало
достаточно запросов к соответствующим доменам:

- **Квота на интернет целиком**: ≥5 DNS-запросов в минуту = активная минута
  (фоновые пинги ОС редкие и порог не пробивают).
- **Квота на категорию** (games / video / social — те же категории, что в
  политиках панели): активная минута = есть запрос к домену категории.
  Ведём словарь доменов-паттернов на категорию (roblox.com, steam*,
  youtube.com, ytimg, tiktok…) — стартовый список зашит в код, живёт рядом
  с SERVICE_CATEGORIES.

Честная оговорка: это приближение (кеш DNS, свернувшееся приложение может
«пинговать» сервер). Для родительского контроля точности достаточно; спорные
случаи решает бонус-кнопка.

### Модель данных (аддитивно, через migrations.py)

- `Quota(id, target_type: group|device, target, category: internet|games|video|social,
  minutes_per_day, days: csv как у Rule, enabled)`
- `QuotaUsage(id, device_id, date, category, minutes)` — счётчики по дням
- `QuotaBonus(id, target_type, target, date, category, minutes, comment)` —
  разовые добавки («+30 минут за уборку»)

### Учёт и применение

- Раз в минуту (тот же reconcile-тик): один запрос журнала AdGuard →
  инкременты QuotaUsage. Дешёво: журнал уже читается watchdog'ом.
- `_desired_state`: квота исчерпана (usage ≥ quota + бонусы за сегодня) →
  - категория → в ag_clients устройства добавляются blocked_services этой
    категории (существующий механизм AdGuard, ничего нового на роутере);
  - интернет целиком → IP в hs-blocked (как пауза, до конца дня).
- Сброс — автоматически сменой даты; таблица QuotaUsage чистится в daily
  cleanup (храним 30 дней — пригодится для дайджеста «сколько сидели»).

### UI и инструменты

- Панель: секция «Квоты» на странице правил (создать/выключить/удалить),
  на странице устройств — прогресс «игры: 1:20 из 2:00».
- Реестр инструментов (для бота и ИИ): `get_quota_status` (чтение),
  `set_quota`, `add_bonus_time` (мутации → как обычно, через кнопку);
  команда бота `/bonus <кто> <минут>`.
- Дайджест дополняется строкой «экранное время за день».

## 3. Страница «время вышло»

### Механика

Сейчас hs-blocked просто дропается — сайты «висят». Делаем перехват HTTP:

- Новый одноразовый скрипт `mikrotik/enable-block-page.rsc` (правило NAT):
  `dst-nat tcp/80 из src-address-list hs-blocked и hs-unknown → 192.168.88.2:8000`.
  Панель NAT-правила по-прежнему не трогает — ставит владелец руками, один
  раз, с обычной проверкой `print where comment~"hs:"`.
- Панель: middleware смотрит IP клиента; если IP сейчас в блокировке/паузе/
  исчерпанной квоте и запрос не к самой панели — отдаёт страницу `/blocked`
  без авторизации: причина (ручная блокировка / пауза до HH:MM / квота:
  осталось 0 из 2:00) и «попроси родителей: /bonus в Telegram».

### Честное ограничение (важно понять до согласования)

Перехватывается только **HTTP (порт 80)**. HTTPS-сайты перенаправить нельзя
без подмены сертификатов (делать не будем) — они продолжат просто не
открываться. На практике страницу увидят: любой http-переход, автопроверки
captive portal у телефонов (иногда), ручной заход на `http://192.168.88.2:8000`.
Это стандартное ограничение всех домашних решений без MITM.

## 4. Портал регистрации неизвестных устройств

Использует тот же NAT-перехват (hs-unknown уже в списке из §3):

- Неизвестное устройство при http-переходе видит не «время вышло», а форму:
  «Чьё это устройство? Имя владельца / комментарий» (без пароля, только из
  локальной сети, rate-limit на IP).
- Заявка = запись `RegistrationRequest(device_id, name, comment, ts)` +
  событие в журнал → бот шлёт владельцу уведомление с кнопками:
  «→ Миша (Дети)», «Гость», «⛔ Блок» — те же кнопки, что у уведомления о
  новом устройстве, плюс текст заявки.
- Работает и при `HS_BLOCK_UNKNOWN=true` (незнакомец заблокирован, но портал
  ему доступен — dst-nat срабатывает раньше drop'а forward-цепочки).

## 5. Анти-MAC-рандомизация

Три уровня, от простого к сложному:

1. **Распознавание**: локально-администрируемый MAC (2-й символ 2/6/A/E) —
   в панели и в уведомлении бота метка «⚠️ случайный MAC» + подсказка
   «выключите „приватный адрес“ для домашней сети на этом устройстве —
   иначе правила будут слетать».
2. **Догадка об объединении**: новое устройство с тем же hostname, что у
   существующего (и то офлайн) → бот предлагает: «Похоже, это снова
   „Планшет Миши“ с новым MAC. Объединить?» — кнопка.
3. **Несколько MAC на устройство**: таблица `DeviceMac(device_id, mac)`
   (аддитивная миграция: backfill из devices.mac; поле devices.mac остаётся
   как «основной» для отображения). discover_devices и enforcement матчат
   lease по любому MAC устройства. «Объединить» = перенос MAC в список
   существующего устройства, дубль удаляется. Инструмент `merge_devices`
   в реестре (мутация → кнопка).

Уровни 1–2 дёшевы; уровень 3 — единственная настоящая схемная эволюция фазы
(ради неё и строили migrations.py). Предлагаю делать все три.

## 6. Порядок работ и риски

Порядок (каждый шаг — отдельный коммит, main всегда зелёный):

1. Квоты: модель + учёт + enforcement + панель + инструменты + тесты.
2. `/blocked` в панели + `enable-block-page.rsc` + инструкция в operations.
3. Портал регистрации (поверх механики §3).
4. MAC: уровни 1–2, затем DeviceMac (уровень 3).

| Риск | Митигируется |
|---|---|
| Ложное «время вышло» из-за фонового DNS | порог ≥5 зап/мин, категории считаются только по своим доменам, бонус-кнопка |
| NAT-перехват сломает что-то живое | правило ставится руками, отдельным .rsc, затрагивает ТОЛЬКО адреса из hs-списков; откат = disable одного правила |
| Неавторизованная страница на панели | /blocked и /register отдают только контент, никаких действий кроме заявки; заявки — rate-limit + длина полей |
| Учёт квот грузит Pi | один GET querylog в минуту, уже читаем его watchdog'ом; инкременты пачкой |
| Миграция DeviceMac на живой базе | аддитивная (CREATE TABLE + backfill), откат не нужен; прогоняется тестом на копии схемы |

Объём: ~4 коммита, сопоставимо с этапами 0+1 вместе. ИИ-инструменты растут
на 4 штуки (get_quota_status, set_quota, add_bonus_time, merge_devices) —
все по прежним правилам (мутации только через кнопку).
